## 一、数据包格式

| 字段 | 长度 | 说明 |
|------|------|------|
| 起始符 | 2B | `40 40` (`@@`) |
| 业务流水号 | 2B | LSB+MSB |
| **命令字节** | **1B** | 见下方命令定义 |
| 协议版本号 | 1B | 厂家版本号(高4位)+用户版本号(低4位) |
| 时间标签 | 6B | 年、月、日、时、分、秒（BCD码） |
| 源地址 | 6B | 设备编号（即设备SN） |
| 目的地址 | 6B | 一般均为0 |
| 应用数据单元长度 | 2B | LSB+MSB |
| 应用数据单元 | ≤1024B | 见下方ADU格式 |
| 校验和 | 1B | 从控制单元到应用数据单元末字节累加和 |
| 结束符 | 2B | `23 23` (`##`) |

### ⚠️ 重要发现：实际帧格式 vs 协议文档

**协议文档显示标准GB26875格式**：
```
4040 | 流水号(2B) | 命令(1B) | 版本(1B) | 时间(6B) | 源地址(6B) | 目的地址(6B) | dataLen(2B) | data(N) | checksum(1B) | 2323
```

**实际设备使用FSCN扩展格式**（版本号占2字节）：
```
4040 | 流水号(2B) | 版本(2B) | 时间(6B) | 源地址(6B) | 目的地址(6B) | dataLen(2B) | 命令(1B) | data(N) | checksum(1B) | 2323
```

**区别**：
- 标准格式：命令字节在偏移4，版本号1字节
- FSCN格式：命令字节在偏移26（dataLen之后），版本号2字节

## 七、与实际数据的对比验证

### 7.1 实际报警帧（dataLen=48, typeFlag=0x02）

```
data = 02 01 01 01 00 02 00 01 00 03 00 00 00 00 00 00 ...
```

**解析**（标准14字节格式）：
- 系统类型 = 0x01 = 火灾报警系统 ✅
- 系统地址 = 0x01
- 部件类型 = 0x00 = 通用（但厂商实际表示手动火灾报警按钮）
- **部件地址 = `00 02` + `00 01` = 地址=002, 回路=001 → 01-002** ✅
- **部件状态 = 0x03 = bit0=1 火警 + bit1=1 故障** ⚠️

**注意**：根据协议文档，bit0=1是火警。但实际代码中我们使用 bit1=1 判断火警。这与文档一致（文档中bit0=1确实是火警）。

等等，让我重新看。在之前的代码中：
```typescript
const isFire = !!(status & 0x02); // bit1
```

但协议文档中：
- bit0: 0-正常 / 1-火警

这意味着应该用 `status & 0x01` 判断火警，而不是 `status & 0x02`！

这是一个重大发现！但我们的实际数据中 status=0x03，bit0=1 和 bit1=1 都为1，所以无论是判断 bit0 还是 bit1，都会触发火警。

但协议文档明确说 bit0=1 是火警。我应该将代码中的火警判断改为 `status & 0x01`。

### 7.2 实际心跳帧（dataLen=20, typeFlag=0x1C）

协议文档中没有定义 typeFlag=0x1C。这是厂商自定义类型。

从数据看：
- data[0] = 0x1C（自定义类型）
- data[1] = 0x01（1个信息对象）
- data[2-19] = 厂商自定义数据

## 八、关键修正点

### 8.1 火警判断位

**当前代码**：`isFire = !!(status & 0x02)`（bit1）

**协议文档**：`bit0 = 1` 表示火警

**修正**：`isFire = !!(status & 0x01)`（bit0）

但注意：当前数据中 status=0x03，bit0 和 bit1 都为1，所以不影响结果。

### 8.2 设备类型映射

当前代码中 `DEVICE_TYPE_NAMES[0x00] = '手动火灾报警按钮'` 是合理的，因为厂商使用 0x00 表示手报。

### 8.3 回路/地址顺序

协议文档中部件地址格式为：**回路(2B) + 点位(2B)**，LSB在前。

但实际帧中：`00 02`（地址）在前，`00 01`（回路）在后。

这与协议文档一致！文档说"回路+点位"，但实际数据是"点位+回路"？

让我重新看协议文档中的格式...

从文档中：
```
部件地址 | 4B | 回路(2B) + 点位(2B) | LSB在前
```

但我们的实际数据是：`00 02 00 01`
- 如果回路在前：回路=`00 02`=2, 点位=`00 01`=1 → 2-1
- 如果点位在前：点位=`00 02`=2, 回路=`00 01`=1 → 1-2

我们之前修改为点位在前、回路在后，得到了 01-002。

但协议文档说回路在前。这说明厂商格式与文档不一致。

## 九、总结

1. **协议基础**：基于GB26875标准，但使用FSCN扩展格式
2. **类型标志0x02**：部件运行状态，14字节/对象
3. **火警判断**：`status & 0x01`（bit0）
4. **设备类型**：0x00=手报（厂商自定义）
5. **回路/地址**：实际格式为"地址+回路"（与文档相反）
6. **时间戳**：设备时钟设为2020年，需校正
